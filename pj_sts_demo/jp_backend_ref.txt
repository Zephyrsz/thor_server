

日语化改造比中文还要顺一些,因为管线里两个关键默认组件(Whisper 系 STT 和 Qwen3-TTS)本身就正式支持日语,真正要动手术的是 STT 选型、分句逻辑和轮次检测的文化适配。我结合刚查到的 2026 年日语语音模型基准测试,逐节点说明。

## VAD:模型不用换,参数要为日语会话习惯调

Silero VAD 是语言无关的,直接沿用。但日语口语有两个特点需要调参应对。第一是**相槌(あいづち)文化**:日语听者会高频插入「はい」「うん」「ええ」这类短应答,如果 barge-in 判定太敏感,机器人每说两句就会被用户的附和打断。对策是把 `min_speech_ms` 从 384 适当上调(500-600ms),并利用 VAD handler 里现成的 `short_segment_merge_ms` 和 `_SHORT_SEGMENT_MIN_FRAGMENT_MS` 机制过滤短促片段——代码注释里明确写了这套机制就是防止"碎片音累积触发误打断"的。第二,日语句尾助词(です/ます/ね/よ/か)是极强的语义端点信号,如果后续做语义 turn detection,日语反而比中英文容易——一个小分类器看实时转写的尾部形态就能高置信度判断"说完了没有",可以支撑更激进的 `min_silence_ms`。

## STT:必换节点,且选型有反直觉的坑

默认的 Parakeet TDT 不支持日语,必须换。仓库内置的 Whisper 系 handler(`whisper`、`faster-whisper`、`lightning-whisper-mlx`、`mlx-audio-whisper`)全部在支持语言列表里包含 `ja`,加 `--language ja` 即可。选型上,2026 年 4 月一份用自然会话音频对 9 个模型做的日语 ASR 基准给出的结论很有参考价值:综合精度和稳定性上 whisper-large-v3-turbo 和 qwen3-asr-1.7b 领先,速度优先选 parakeet-tdt-0.6b-v3(欧洲语言向,日语不适用),日语媒体领域特化选 reazonspeech。

一个反直觉的坑:日语社区很推的 Kotoba-Whisper(用 ReazonSpeech 720 万日语电视音频片段蒸馏 whisper-large-v3,速度快 6.3 倍且 CER 相当)看起来是完美选择,但该基准明确把 kotoba-whisper-v2.0 列为"自然会话场景应避免"的模型——因为它的训练数据是电视播音,对口语的语气词、犹豫、不完整句表现差,而语音助手恰恰全是自然会话。同样,ReazonSpeech 系列在多说话人重叠时倾向于截断后半段,远场多人环境要慎用。

所以我的推荐是:**首选 `--stt faster-whisper --faster_whisper_stt_model_name large-v3-turbo --language ja`**(CTranslate2 int8 量化后单句转写几十毫秒级,精度天花板);如果想要更小更快且愿意写一个新 handler,qwen3-asr-1.7b 值得评估——继承 `base_stt_handler` 实现 `setup/process` 即可接入。另外日语评测要看 CER 而不是 WER(日语无空格分词,WER 计算不公平),自建测试集时注意这点。

## LLM:选日语能力强的模型 + 敬语控制

架构上零改动(OpenAI 兼容接口),纯粹是模型选择问题。你 L20 上跑 vLLM 的话:Qwen3 系列日语能力可用但不是最强;日语特化的开源选项主要是 Llama-3.1-Swallow(东工大,日语续训)、ELYZA、CyberAgent 的 CALM、SB Intuitions 的 Sarashina 系列,选型时建议对照 Nejumi 日语 LLM 排行榜的最新结果,这个领域迭代很快。语音场景的额外要求写进 system prompt:**指定敬语等级**(です・ます体还是 casual)、回答简短口语化、数字用日语读法习惯表达。前面说过的 prefix caching、关 reasoning 等优化对日语完全同样适用。

## 分句逻辑:日语部署必改的一行

`base_openai_compatible_language_model.py` 里的 nltk `sent_tokenize` 默认按英文规则,对日语的「。」「!」「?」以及引号「」不可靠,会导致 TTS 收到超长片段,首句延迟直接劣化。最小改动是把该处替换成按 `。!?\n` 切分的正则(注意「?」和半角 `?` 都要覆盖);更讲究的用 bunkai 或 konoha 这类日语分句库,能正确处理引号内句号和颜文字。另外 `_flush` 里 `" ".join(batch)` 会在日语句子间插入空格,对 TTS 一般无害但严谨起见可以按语言条件去掉。

## TTS:默认 Qwen3-TTS 直接可用,日语特化引擎是备选升级

好消息是零改动可跑:仓库的 `qwen3_tts_handler` 语言映射表里已有 `ja/jp → japanese`,启动时加 `--qwen3_tts_language ja` 即可。Qwen3-TTS 是 2026 年 1 月开源的多语言模型,支持 3 秒参考音频声音克隆、自然语言 VoiceDesign,流式生成首音延迟约 97ms,Apache-2.0 商用无碍,对语音 agent 是目前的均衡最优解。内置的 Kokoro(82M)也有日语音色(jf_alpha 等),可以按我上次说的"双速策略"做首句抢跑。

如果对日语音质有更高要求,日语特化引擎是升级方向:Style-BERT-VITS2 和 AivisSpeech 在**音调重音(ピッチアクセント)和汉字读音消歧**上通常优于多语言模型——这是日语 TTS 特有的难点(「重複」读 ちょうふく 还是 じゅうふく、助数词变音等),多语言模型偶尔会读错。VOICEVOX 则是 CPU 即可运行的轻量选项,角色音色丰富,但多数角色商用需要署名,风格也偏动漫向,面向企业场景要斟酌。这些引擎接入都很简单:VOICEVOX 有本地 REST API,写一个 50 行的 handler 就能挂进 `lm_processed_queue`。TTS 前的文本规范化在日语里同样重要:数字转假名读法、日期格式、外来语罗马字的处理,放在 LMOutputProcessor 里做。

## 一条可直接起跑的配置

在你的 CUDA 服务器上,日语版的启动大致是:

```bash
python s2s_pipeline.py \
  --stt faster-whisper \
  --faster_whisper_stt_model_name large-v3-turbo \
  --language ja \
  --llm chat-completions \
  --chat_completions_api_base http://<vllm-host>:8000/v1 \
  --chat_completions_model_name <日语强的模型> \
  --tts qwen3 --qwen3_tts_language ja \
  --min_speech_ms 550 --short_segment_merge_ms 200
```

改动优先级排序:**分句正则(十分钟,收益立竿见影)→ STT 换 large-v3-turbo(必改)→ VAD 相槌调参(实测迭代)→ system prompt 敬语规范 → 日语特化 TTS 评估(可选升级)**。上线前建议用日常会话(而非朗读文本)录一批测试音频跑 CER 和端到端延迟,日语的自然会话和朗读语料之间模型表现差异比中英文更大——上面 Kotoba-Whisper 的教训就是例证。




基础模型优化


结合 2026 年上半年的日语基准数据,40B 以下、llama.cpp 生态友好这个约束下,答案其实相当收敛:**开源小中型模型的日语能力目前是 Qwen3 系及其日语强化版的天下**。多个日语评测(Nejumi 4、Shisa 的 Rakuda/Tengu 等)得出的共同结论是"小型模型做日语,Qwen3 事实上是唯一选择",而且日本本土模型没有一个进入 Nejumi 4 综合 Top 50——想要日语强的开源模型,反而要在 Qwen 系里挑。

## 首选:Qwen3-Swallow-30B-A3B-RL-v0.2(东京科学大 Swallow 团队)

这是我给语音 agent 场景的第一推荐。Swallow 项目对 Qwen3 做了日语继续预训练(CPT)+ SFT + RL 的完整日语化,产出了 8B、32B 和 30B-A3B(MoE)三个规格,30B-A3B 还有官方 AWQ-INT4 版本。其中 Qwen3 Swallow 32B 在 2026 年 2 月被报告为 32B 以下开源模型中日语任务性能最高。

对你的场景,我更推 **30B-A3B(MoE)版本而不是 32B dense**,理由是延迟:MoE 每 token 只激活 3B 参数,decode 速度接近 3B 小模型,但质量接近 32B——这正是语音对话要的"又快又好"。Qwen3 架构 llama.cpp 完整支持,社区 GGUF(mmnga 等日本量化作者通常第一时间跟进 Swallow 系列)直接可用,Q4_K_M 约 18GB,单张 L20 绰绰有余。备选顺位:如果实测 MoE 版在你的任务上质量不够,退回 **Qwen3-Swallow-32B-RL-v0.2**(dense,Q4 约 20GB);如果想压显存,**Qwen3-Swallow-8B-RL-v0.2** 是同系小杯。

## 值得关注但有保留的选项

**Qwen3.5-27B**:Nejumi 4 上以 0.8049 成为首个突破 0.80 的开源中型模型,达到三个月前商用 API 顶级水平,纸面日语最强。保留意见是它是新架构,选用前要确认你手头 llama.cpp 版本已支持(新架构在 llama.cpp 的支持通常滞后数周),以及是否有可靠 GGUF。支持了就值得 A/B。

**Nemotron Nano 9B v2 Japanese**:Nejumi 4 上 10B 以下第一(0.7111),Transformer-Mamba 混合架构,吞吐可达同尺寸开源模型的 6 倍,NVIDIA 专为日本市场做的。但混合 SSM 架构在 llama.cpp 上的支持和优化程度需要验证,更适合留在 TensorRT-LLM/vLLM 栈里;llama-server 部署优先级放低。

**Gemma 3 27B / Gemma 4 26B-A4B**:日语不差(Gemma 4 的 26B-A4B 在日语任务测到 96.4%,超过 120B 的 Nemotron 3 Super),QAT int4 显存友好,作为对照组可以,但日语专项上一般不敌 Qwen 系。

## 两个部署层面的关键注意点

第一,**关闭思考模式**。Qwen3/Swallow RL 版默认带 thinking,语音场景 TTFT 会爆炸。llama-server 用 `--jinja` 加载聊天模板后,在管线的请求侧通过 `chat_template_kwargs: {"enable_thinking": false}` 或在 system prompt 尾部加 `/no_think` 关闭,务必实测确认输出里没有 `<think>` 段——否则还需要在 LMOutputProcessor 里加一道剥离兜底。

第二,**中文混入问题**。这是 Qwen 系做日语的已知缺陷:ABEJA 验证过 Qwen2.5-7B 日语输出中有 0.9% 概率混入中文,日语微调可降到 0.2%,但这是共享 token 导致的结构性问题,无法完全消除。选 Swallow 日语化版本本身就是最有效的缓解;残余风险在 TTS 前的文本处理里再加一道简单的汉字白名单/中文标点检测,发现混入就丢弃该句或触发重生成,语音场景一个中文词被日语 TTS 硬读出来非常刺耳。

## 参考启动命令

```bash
llama-server \
  -m Qwen3-Swallow-30B-A3B-RL-v0.2-Q4_K_M.gguf \
  -ngl 99 -fa on \
  -c 16384 --jinja \
  --cache-reuse 256 \
  -np 2 \
  --host 0.0.0.0 --port 8000
```

`--cache-reuse` 让 llama-server 复用请求间的公共前缀 KV cache——语音对话每轮都带完整历史,这个开关等价于我之前说的 prefix caching,对 TTFT 收益极大;`-np` 按并发会话数设。如果最终选了 32B dense 版,再加 `--model-draft Qwen3-0.6B` 做投机解码把 decode 提速。管线侧对应 `--chat_completions_api_base http://<host>:8000/v1`,model name 随便填(llama-server 单模型不校验)。

建议的验收方法:拿 20 条你真实场景的日语口语 query,同时起 30B-A3B 和 32B dense 各跑一遍,记录 TTFT、每句敬语规范性和中文混入次数——这三个指标比任何排行榜都更能决定该用哪个。
